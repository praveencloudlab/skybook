# 1. Architecture

```
Controller
    │
    ▼
BookingFacade   ← only layer that knows other services exist
    │
    ├── FlightServiceClient → FlightServiceFeignClient (@FeignClient → flight-service)
    ├── BookingService (owns the Booking aggregate)
    │       │
    │       ▼
    │   BookingRepository, BookingPassengerRepository, ...
    │
    ├── Domain Services: PnrGenerator, BookingStateMachine, BookingValidator,
    │                     FareCalculator, SeatAssignmentStrategy
    │
    └── BookingEventProducer → Kafka (skybook.booking.events) → notification-service
```

`BookingFacade` wraps only `createBooking` / `confirmBooking` / `cancelBooking` — the three operations with something to orchestrate (flight validation and/or a notification event). Everything else is called directly from `BookingController` → `BookingService`.

`BookingFacade` methods are deliberately **not** `@Transactional`: by the time control returns from a `BookingService` call (which is `@Transactional`), that transaction has committed, so publishing to Kafka afterward is equivalent to `@TransactionalEventListener(AFTER_COMMIT)` without the extra machinery.

# 3. Enums & State Machines

## 3.1 `BookingStatus`

`CREATED`, `CONFIRMED`, `CANCELLED`, `COMPLETED`

Golden transition table (`BookingStateMachine.BOOKING_TRANSITIONS`):

| From \ To | CREATED | CONFIRMED | CANCELLED | COMPLETED |
|---|---|---|---|---|
| CREATED | — | ✅ | ✅ | ❌ |
| CONFIRMED | ❌ | — | ✅ | ✅ |
| CANCELLED | ❌ | ❌ | — | ❌ |
| COMPLETED | ❌ | ❌ | ❌ | — |

Moving to `CANCELLED` cascades every passenger's `CheckInStatus → CLOSED`.

## 3.2 `PaymentStatus`

`PENDING`, `PAID`, `FAILED`, `REFUNDED`

| From \ To | PENDING | PAID | FAILED | REFUNDED |
|---|---|---|---|---|
| PENDING | — | ✅ | ✅ | ❌ |
| FAILED | ✅ | ❌ | — | ❌ |
| PAID | ❌ | — | ❌ | ✅ |
| REFUNDED | ❌ | ❌ | ❌ | — |

Moving to `PAID` stamps `paidAt = now()`.

## 3.3 `CheckInStatus` (per passenger)

`NOT_OPEN`, `OPEN`, `CHECKED_IN`, `BOARDED`, `NO_SHOW`, `CLOSED`

| From \ To | NOT_OPEN | OPEN | CHECKED_IN | BOARDED | NO_SHOW | CLOSED |
|---|---|---|---|---|---|---|
| NOT_OPEN | — | ✅ | ❌ | ❌ | ❌ | ✅ |
| OPEN | ❌ | — | ✅ | ❌ | ✅ | ✅ |
| CHECKED_IN | ❌ | ❌ | — | ✅ | ✅ | ✅ |
| BOARDED | ❌ | ❌ | ❌ | — | ❌ | ✅ |
| NO_SHOW | ❌ | ❌ | ❌ | ❌ | — | ✅ |
| CLOSED | ❌ | ❌ | ❌ | ❌ | ❌ | — |

## 3.4 `TravelClass` / `FareType`

`TravelClass`: `ECONOMY`, `PREMIUM_ECONOMY`, `BUSINESS`, `FIRST`
`FareType`: `SAVER`, `FLEXI`, `PREMIUM`

## 3.5 `BookingHistoryField`

`BOOKING_STATUS`, `PAYMENT_STATUS`, `CHECK_IN_STATUS` — which of the 3 state machines a `BookingHistory` row records.

## 3.6 Cross-machine invariants (`BookingValidator`)

- Check-in only allowed if `BookingStatus = CONFIRMED` **and** `PaymentStatus = PAID`.
- Refund (`PaymentStatus → REFUNDED`) only allowed if `BookingStatus = CANCELLED`.
- A passenger's passport must expire strictly after the flight's departure date.

# 4. Domain Services

| Class | Responsibility | I/O? |
|---|---|---|
| `PnrGenerator` | Generates a PNR candidate: `"SB"` + 4 random chars from `23456789ABCDEFGHJKMNPQRSTUVWXYZ` (excludes `0/O/1/I/L`). No uniqueness check — that's the caller's job. | None (`SecureRandom` only) |
| `BookingStateMachine` | `canTransitionBooking/Payment/CheckIn(from, to)` + `transitionBookingStatus/PaymentStatus/CheckInStatus(...)`, which validate, mutate the entity, and append a `BookingHistory` row. | None — mutates entities handed to it |
| `BookingValidator` | `validateCheckInAllowed`, `validateRefundAllowed`, `validatePassportValidForTravel` — cross-machine invariants (§3.6). | None |
| `FareCalculator` | `calculateFare(travelClass, fareType)` — placeholder fixed base fares (Economy $100 → First $700) × fare-type multiplier (Saver 0.85 / Flexi 1.00 / Premium 1.25), scaled to 2dp. Flagged as a placeholder pricing model pending a real pricing source. | None |
| `SeatAssignmentStrategy` (interface) / `ManualSeatAssignmentStrategy` | `resolveSeatNumber(passenger, requestedSeatNumber)` — v1 requires the caller to supply a seat, uppercases it, rejects blank. No auto-assignment. | None |

`generateUniquePnr()` (in `BookingServiceImpl`) retries `PnrGenerator.generateCandidate()` against `BookingRepository.existsByBookingReference(...)` up to **10** attempts, throwing `IllegalStateException` if all collide.

# 7. REST API Reference

Base path: `/api/bookings`

| Method | Path | Handler | Description |
|---|---|---|---|
| `POST` | `/api/bookings` | `bookingFacade.createBooking` | Validates flight via Feign, checks seat availability, computes fares, generates PNR, saves aggregate, publishes `CREATED` event. Returns `201`. |
| `GET` | `/api/bookings/{id}` | `bookingService.getBookingById` | By internal id. |
| `GET` | `/api/bookings/reference/{pnr}` | `bookingService.getBookingByReference` | By PNR. |
| `GET` | `/api/bookings` | `bookingService.getAllBookings` | List all. |
| `GET` | `/api/bookings/search` | `bookingService.searchBookings` | Query params: `bookingReference`, `flightId`, `passengerName`, `passportNumber`, `bookingStatus`, `paymentStatus`, `travelDate`, `bookingDate`, `email`, `phone` — all optional, in-memory filtering over `findAll()`. |
| `PATCH` | `/api/bookings/{id}/confirm` | `bookingFacade.confirmBooking` | Simulates payment success (`PENDING→PAID`), then `CREATED→CONFIRMED`. Publishes `CONFIRMED` event. |
| `PATCH` | `/api/bookings/{id}/cancel` | `bookingFacade.cancelBooking` | Body: `CancelBookingRequest` (optional). `→CANCELLED`, cascades check-in closure, refunds if paid. Publishes `CANCELLED` event. |
| `PATCH` | `/api/bookings/{id}/complete` | `bookingService.completeBooking` | `CONFIRMED→COMPLETED`. |
| `PATCH` | `/api/bookings/{id}/passengers/{passengerId}/check-in` | `bookingService.checkInPassenger` | `passengerId` = `BookingPassenger.id`. Auto-opens the check-in window if `NOT_OPEN`, then `→CHECKED_IN`. |
| `PATCH` | `/api/bookings/{id}/passengers/{passengerId}/board` | `bookingService.boardPassenger` | `CHECKED_IN→BOARDED`. |

Swagger UI: `http://localhost:8083/swagger-ui.html`; OpenAPI JSON at `/api-docs`.

# 9. Kafka Event Flow

- `FlightServiceClient` → `FlightServiceFeignClient` (`@FeignClient(name="flight-service", url="${flight-service.base-url}")`) → `GET /api/flights/{id}` on flight-service, translated to `FlightNotFoundForBookingException` (404) / `FlightServiceUnavailableException` (any other Feign failure).
- `BookingEventProducer` publishes `com.skybook.praveen.common.event.BookingEvent` (type `CREATED`/`CONFIRMED`/`CANCELLED`) to topic `skybook.booking.events` (`KafkaTopics.BOOKING_EVENTS`) via `KafkaTemplate<String, BookingEvent>`. Skipped with a warning log if the booking has no contact.
- `notification-service`'s `BookingEventConsumer` (`@KafkaListener`) maps `BookingEvent → EmailEvent` (type `BOOKING_NOTIFICATION`) and hands off to the existing `EmailService`.
- Publish happens *after* the wrapping `BookingService` `@Transactional` call returns (i.e., after commit) since `BookingFacade` methods are not themselves transactional — see §1.

# 12. Known Limitations / Deferred Work

- **`travelDate` search filter is accepted but not applied** — filtering by flight departure date would require calling `flight-service` per search result, which isn't wired up for the search path (only for create).
- **No real Payment Service** — `confirmBooking` simulates a successful payment directly.
- **No real seat map / auto-assignment** — caller must supply a seat number.
- **No authentication/authorization enforcement** — Spring Security is wired in permit-all mode; JWT validation is a follow-up once auth-service tokens can be verified here.
- **Testcontainers dependencies are present but unused** — all current tests are pure unit/mock-based; no DB-backed integration test exists yet.
- **No transactional outbox** — Kafka publish-after-commit via non-transactional facade methods is the current mitigation for the dual-write problem; revisit if delivery guarantees need to be stronger.
- **Single flight per booking** — no multi-segment/round-trip itineraries yet.
